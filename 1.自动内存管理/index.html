JVM►《深入理解Java虚拟机》读书笔记

1.自动内存管理

自动内存管理

1 运行时数据区

1.1 程序计数器

  1. 一块较小的内存空间
  2. 当前线程所执行的字节码的行号指示器，字节码解释器工作时就是通过改变这个计数器的值来选取下一条需要执行的字节码指令
  3. 因为多线程间涉及到线程的切换，为了线程切换后能恢复到正确的执行位置，每条线程都需要有一个独立的程序计数器，各条线程之间计数器互不影响，独立存储，这块内存区域为"线程私有"内存。
  4. 如果线程执行的是一个Java方法，则程序计数器中记录的是当前执行的字节码指令的地址；如果执行的是一个Native方法，则计数器的值为空(Undefined)
  5. 唯一一个没有规定任何OutOfMemoryError的区域

1.2 Java虚拟机栈

  1. 也是线程私有的，生命周期与线程相同。
  2. 当每个方法被执行时，JVM都会创建一个栈帧，存储局部变量表、操作数栈、动态链接、方法出口等信息
  3. 如果线程请求的栈深度大于虚拟机允许的深度，则抛出StackOverflowError；如果Java虚拟机栈的容量可以动态扩展，则当栈扩展到无法申请足够内存时抛出OutOfMemoryError。

1.2.1 局部变量表

  1. 存放了编译器可知的基本数据类型(boolean、byte、char、short、int、float、long、double)、对象引用(reference类型，它并不等同于对象本身，可能是一个指向对象起始地址的引用指针，也可能是指向一个代表对象的句柄或者其他与此对象相关的位置)、 returnAddress类型（指向了一条字节码指令的地址）
  2. 以上的数据类型在局部变量表中的存储空间是以局部变量槽(slot)为单位的，除long类型和double类型会占用两个slot外，其余类型都只会占用一个slot。局部变量表所需要的空间在编译期间就已经确认，其中slot的个数在编译期间确定，而每个slot占用的内存空间由JVM的实现确定。

1.2.2 操作数栈

  1. 用于临时存储操作数和返回结果

  2. int a = 1;
int b = 2;
int c = a + b;

    在执行加法操作时，Java 虚拟机会将 a 和 b 这两个操作数从局部变量表中加载到操作数栈中，然后执行加法操作，将结果 c 存储到操作数栈中。

1.2.3 动态链接

  1. 指在程序运行时（而不是编译时）通过符号引用来动态地确定调用的方法实现或者访问的字段的内存地址的过程，这样做的好处是可以实现类共享（即多个类加载器加载同一个类时只需要在内存中加载一份），从而降低程序占用的内存。

1.3 本地方法栈

  1. 对于本地方法栈中方法使用的语言、使用方式与数据结构并没有任何强制规定。
  2. 如果线程请求的栈深度大于虚拟机允许的深度，则抛出StackOverflowError；如果Java虚拟机栈的容量可以动态扩展，则当栈扩展到无法申请足够内存时抛出OutOfMemoryError。

1.4 Java堆

  1. 几乎所有的对象实例都在堆上分配内存
  2. “新生代”“老年代”“永久代”“Eden 空间”“From Survivor 空间”“To Survivor 空间”等名词仅仅是一种经典分代，在多年前的HotSpot虚拟机中的所有垃圾收集器都采用的这种分代，但时至今日已经有很多不采用分代设计的新的垃圾收集器。
  3. 可以处于物理上不连续的内存空间中，但在逻辑上是连续的。
  4. Java 堆中没有内存完成实例分配，并且堆也无法再扩展时，Java 虚拟机将会抛出 OutOfMemoryError

1.5 方法区

  1. 线程共享的内存区域。用于存储已被虚拟机加载的类型信息、常量、静态变量、即时编译器编译后的代码缓存等数据。
  2. 虽然《JVM规范》中将方法区描述为堆的一个逻辑部分，但是还是和堆区分开好。
  3. 关于方法区和永久代的问题，永久代是HotSpot中《JVM规范》关于方法区的实现，并非是一个等价概念。JDK7时HotSpot把原本存在永久代的字符串常量池、静态变量等移出，JDK8时废除永久代，使用在本地内存中实现的元空间（Metaspace）代替，并将JDK7中永久代中剩余内容全部移到元空间中。
  4. 如果方法区无法满足新的内存分配需求时，抛出OutOfMemoryError

1.5.1 运行时常量池

  1. 方法区的一部分，Class 文件中除了有类的版本、字段、方法、接口等描述信息外，还有一项信息是常量池表（Constant Pool Table），用于存放编译期生成的各种字面量与符号引用，这部分内容会在类加载后存储到运行时常量池中。
  2. 并不要求常量一定只有编译期才能产生，也就是说，并非预置入 Class 文件中常量池的内容才能进入方法区运行时常量池，运行期间也可以将新的常量放入池中，这种特性被开发人员利用得比较多的便是 String 类的 intern()方法。
  3. 当常量池无法再申请到内存时会抛出 OutOfMemoryError。

1.6 直接内存

  1. 比如NIO中DirectByteBuffer对象可以直接对直接内存的引用进行操作。
  2. 当需要动态扩展时超出了本机总内存的大小时，抛出OutOfMemoryError。

1.7 执行引擎

  1. 分为解释器和即时编译器两个部分，解释器将字节码文件逐行解释为机器码，由于每一次执行都需要解释对应字节码为机器码，效率较低；而即时编译器是将字节码文件一次性编译为机器码文件，效率较高。当方法被多次调用时，JVM会使用即时编译器编译该字节码，从而提高执行效率。

2 HotSpot虚拟机中的对象

2.1 普通对象的创建（不包括数组和Class对象）

普通对象的创建过程

  1. JVM在遇到一条字节码new指令时，先检查该指令的参数能否在常量池中定义到一个类的符号引用，并检查该符号引用代表的类是否已经被加载、解析、初始化，如果没有，则必须先执行相应的类加载流程。
  2. 为新生对象分配内存。对象所需内存在类加载后即可完全确定，故只需要将一块确定大小的内存从堆中划分即可。这部分需要后续的垃圾回收详细了解。
  3. 内存分配完成之后，虚拟机必须将分配到的内存空间（但不包括对象头）都初始化为零值，如果使用了 TLAB （Thread Local Allocation Buffer，本地线程分配缓存）的话，这一项工作也可以提前至 TLAB 分配时顺便进行。
  4. 设置对象的对象头信息。

2.2 对象的内存布局

  1. 在HotSpot中，对象在堆内存中的存储可以分为：对象头（Header）、实例数据（Instance Data）、对齐填充（Padding）

  2. 对象头分为运行时数据（MarkWord）、类型指针

    1. MarkWord

      1. 存储HashCode、GC年龄分代、锁状态标志、线程持有的锁、偏向线程ID、偏向时间戳等，结构为BitMap

      2. 长度在 32 位和 64 位的虚拟机（未开启压缩指针）中分别为 32 个bit和 64 个bit

      3. 其中存储的信息及其所占大小随对象加锁状态变化而变化

    2. 类型指针

      1. 指向对象类型元数据的指针

  3. 对象中所有的各种类型的字段内容的存储顺序序会受到虚拟机分配策略参数（-XX：FieldsAllocationStyle 参数）和字段在 Java 源码中定义顺序的影响。HotSpot 虚拟机默认的分配顺序为 longs/doubles、ints、shorts/chars、bytes/booleans、oops（Ordinary Object Pointers，OOPs），可以看出相同宽度的字段总是被分配到一起存放。如果 HotSpot 虚拟机的+XX：CompactFields 参数值为true（默认就为 true），那子类之中较窄的变量也允许插入父类变量的空隙之中，以节省出一点点空间。

  4. 对齐部分。HotSpot中要求对象的起始地址必须是8Byte的整数倍，故需要对齐。

2.3 对象的访问定位

  1. JVM会通过栈上栈帧中的reference数据来操作堆中具体对象。

  2. reference数据的实现并没有规定，故主流的方式有两种：句柄、直接指针

    1. 堆中可能划分出一块内存作为句柄池，reference中存储句柄地址，句柄中包含对象实例数据和类型数据各自的具体地址信息。

    2. reference中存放的直接就是对象实例数据的指针

    3. 使用句柄来访问的最大好处就是 reference 中存储的是稳定句柄地址，在对象被移动（垃圾收集时移动对象是非常普遍的行为）时只会改变句柄中的实例数据指针，而 reference 本身不需要被修改。

    4. 使用直接指针来访问最大的好处就是速度更快，它节省了一次指针定位的时间开销。

3 垃圾收集器和内存分配策略

3.1 垃圾回收算法

3.1.1 标记算法

3.1.1.1 引用计数算法
  1. 在对象中添加一个引用计数器，每当有一个地方引用它时，计数器值就加一；当引用失效时，计数器值就减一；任何时刻计数器为零的对象就是不可能再被使用的。
  2. 无法解决两个对象相互引用的回收问题
3.1.1.2 可达性分析算法
  1. 通过一系列称为“GC Roots”的根对象作为起始节点集，从这些节点开始，根据引用关系向下搜索，搜索过程所走过的路径称为“引用链”（Reference Chain），如果某个对象到 GC Roots 间没有任何引用链相连，或者用图论的话来说就是从 GC Roots 到这个对象不可达时，则证明此对象是不可能再被使用的。
  2. 在Java中，固定可以作为GC Roots的对象如下：
    1. 在方法区中类静态属性引用的对象
    2. 在方法区中常量引用的对象
    3. 本地方法栈中JNI（Java Native Interface）引用的对象
    4. JVM内部的引用，如基本数据类型的Class对象，常驻的异常对象，系统类加载器
    5. 所有被同步锁持有的对象
    6. 反映 Java 虚拟机内部情况的 JMXBean、JVMTI 中注册的回调、本地代码缓存等。

3.1.2 标记-清除算法

  1. 标记所有存活对象或者非存活对象，之后使用清除算法对所有未标记的或者已标记的对象进行回收

  1. 主要有两个缺点：一是执行效率不稳定。如果堆中有大量的对象，且其中大部分需要被回收，那么标记和清除两个过程都会变长，效率降低；二是在内存回收过后内存空间会碎片化，从而导致之后分配较大对象时无法找到连续的内存空间再一次触发内存回收。

3.1.3 标记-复制算法

  1. 将内存区域划分为等大小的两块，存储对象时存储至其中的一块内存区域，当需要回收时，标记所有的存活对象，将存活对象复制进另外一块内存空间，之后将原来的内存空间一口气回收。

![标记-复制算法](./1.自动内存管理/image-20230406102337281.png)

  1. 首先，这种算法的最大问题是每次都会有一半的内存空间会浪费；其次，如果每次垃圾回收后的存活对象非常多，那么复制消耗的资源也会相当大。故这种算法一般不会用在老年代中。

  2. Appel式回收

    1. 因为大多数的新生代对象都活不过第一次垃圾回收，所以并不需要按照1:1的比例来划分新生代的内存空间。Appel式回收的具体做法是，将整个新生代划分为一块Eden空间和两块Survivor空间，每次分配内存时至分配至Eden和一块Survivor中，当需要回收时，将Eden和Survivor中存活对象复制进另一个Survivor中，然后将Eden和此前的Survivor直接回收。Eden和Survivor的大小比为8:1，所以每次新生代中可用的内存空间为90%。
    2. 当然，如果某一次存活对象的内存大小超过了10%，就需要依赖其他内存区域（大部分是老年代）进行分配担保。如果另外一块 Survivor 空间没有足够空间存放上一次新生代收集下来的存活对象，这些对象便将通过分配担保机制直接进入老年代，这对虚拟机来说就是安全的。
    3. HotSpot中的Serial、ParNew等新生代垃圾收集器都是采用的这种策略来设计新生代的内存布局。

3.1.4 标记-整理算法

  1. 针对老年代对象的存活特征的算法。标记过程依旧是标记出所有的存活对象，然后让所有存活对象的内存空间向内存空间的一端移动，之后直接清理掉边界之外的内存。

  1. 首先，老年代的每次回收必然有大量的对象存活，那么移动存活对象就是一项相当耗费资源的工作，并且这种对象的移动操作必须要暂停用户的程序才能进行（Stop The World，STT）。但是如果像标记清除算法的话，堆中内存的碎片化问题只能通过更加复杂的内存分配器和内存访问器来解决了，而内存的访问和分配是用户程序频率最高的动作，所以说即使节省了对象移动操作耗费的时间，但是总的吞吐量还是降低了。在HopSpot中关注吞吐量的Parallel Scavenge收集器基于的是标记-整理算法，关注延迟的CMS收集器是基于标记-清除算法。

  2. 为什么在对象移动操作过程中必须STT：不管reference的实现是句柄还是直接指针，都需要一个指针指向堆中对象的内存地址，而移动对象的过程中必然会发生指针的变化，如果此时不暂停用户线程，导致使用了已回收对象或者空的内存区域，那么就有可能使程序崩溃。

  3. 既然标记-清除和标记-整理算法各有优劣，那么便可以将二者结合来使用，CMS收集器在空间碎片化过多时便是采用以下方法：大多数时候都采用标记-清除算法来回收内存区域，暂时容忍内存碎片化，当碎片化过多的影响到对象的分配时，采用标记-整理算法来处理碎片化问题。

3.2 HotSpot的算法细节实现

3.2.1 根节点枚举

  1. 首先从可达性分析算法中的GCRoots集合中找引用链开始。固定可以作为GCRoots的对象主要是全局性的引用（常量、类静态属性）和执行上下文（栈帧中的本地变量表）。虽然在可达性分析算法中耗时最长的查找引用链的过程已经可以和用户线程一起并发执行，但是根节点枚举的这一过程还是必须在一个能够保障一致性的快照中执行，所以在根节点枚举的过程中，所有的垃圾收集器都是需要停顿的。
  2. 对于每一个方法，HotSpot都会生成一个OopMap（Ordinary Object Pointer Map）与之对应，所谓的OopMap就是存储当前方法中有哪些引用，分别在哪些位置。这样收集器在扫描时就不需要一个不漏的从方法区等GC Roots等开始寻找了。

3.2.2 安全点

  1. 在OopMap的协助下，HotSpot可以很快的完成GCRoots的枚举，但是只要引用关系变化，就会导致OopMap中的内容发生变化，而这样的指令是相当多的，频繁的修改OopMap不仅会花费较多的时间，也会需要大量额外的存储空间。

  2. HotSpot只会在特定位置（安全点）记录OopMap中的信息，所以也就要求用户程序必须执行到安全点之后才可以暂停。因此安全点的选取很重要，不仅不会间隔时间太长导致垃圾收集器等待时间过长，也不会太过频繁导致资源开销很大。安全点的选取基本上以“是否具有让程序长时间执行的特征”为标准来选定，因为每条指令执行的时间都非常短暂，程序不太可能因为指令流长度太长这样的原因而长时间执行，“长时间执行”的最明显特征就是指令序列的复用，例如方法调用、循环跳转、异常跳转等都属于指令序列复用，所以只有具有这些功能的指令才会产生安全点。

  3. 所谓的安全点就是，在某一个位置上，线程内部的引用关系不会发生改变，在这个位置上，可以进行垃圾收集、编译等操作。

  4. 还有一个问题，如何在垃圾收集时让除执行JNI调用的以外的所有线程都跑到最近的安全点，这里就有两种方案：抢先式中断和主动式中断。

    1. 抢先式中断是在垃圾收集发生时，系统首先把所有用户线程都中断，如果有用户线程不在安全点上，则恢复该线程的执行，直到跑到安全点上再中断。现在几乎没有JVM的实现会采用这种方式响应GC事件。

    2. 主动式中断是指当垃圾收集需要中断线程时，只设置一个标志位，每个线程执行过程中不断的去轮询这个标志，一旦中断标志为true时就在最近的安全点主动挂起，轮询标志的地方和安全点是重合的，还有需要在堆上分配空间的地方也是轮询标志的地方，因为需要检查是否即将发生垃圾收集，避免没有足够的内存分配对象。

    3. 由于轮询操作会在代码中频繁出现，所以必须足够高效，HotSpot使用内存保护陷阱的方式。下面代码清单中的test 指令就是 HotSpot 生成的轮询指令，当需要暂停用户线程时，虚拟机把 0x160100 的内存页设置为不可读，那线程执行到 test 指令时就会产生一个自陷异常信号，然后在预先注册的异常处理器中挂起线程实现等待，这样仅通过一条汇编指令便完成安全点轮询和触发线程中断了。

    4. 内存保护陷阱（Memory Protection Fault）通常指的是操作系统中的一种异常，也叫做“段错误”（Segmentation Fault），通常是因为程序试图访问未经授权的内存区域，或者因为内存访问越界等原因导致的。当发生内存保护陷阱时，操作系统会中断当前程序的执行，并将控制权转移到异常处理器中，从而避免程序继续执行可能导致更严重后果的代码。

3.2.3 安全区域

  1. 线程可以在需要暂停的时候主动在安全点挂起，但是如果此时的线程是blocked状态或者是sleep状态时，线程无法主动的相应JVM的暂停请求，这个时候就需要安全区域。
  2. 安全区域可以确保在一段代码片段内，该线程内对象的引用关系不会发生变化，所以在这一段区域内的任何位置开始垃圾收集都是安全的。
  3. 当线程执行到安全区域的代码时，首先会标识自己已经进入到了安全区域，JVM在需要发起垃圾收集时就不需要管已经在安全区域的线程了；当线程需要离开安全区域时，会检查JVM是否已经完成了如根节点枚举之类的需要暂停用户线程的操作，如果JVM没有完成，则该线程会继续等待直到完成，否则会继续执行。

3.2.4 记忆集、卡表

  1. 在分代收集理论中，会出现跨代引用的问题，为了避免跨代引用而导致将整个老年代加入GC Roots的扫描范围中（假如需要回收新生代，且此时新生代中的部分对象被老年代对象引用，那么此时就需要扫描整个老年代来判断哪些新生代对象不可达），垃圾收集器在新生代中建立了记忆集的数据结构（Remembered Set）。

  2. 记忆集是一种用于记录非收集区域指向收集区域的指针集合，最简单的实现便是一个非收集区域中所有包含跨代引用的对象数组，但是这样的方式空间占用和维护成本非常高，而在实际的垃圾收集的场景中，收集器只需要通过记忆集判断某一块非收集区域是否存在指向了收集区域的指针即可，所以记忆集的实现可以选择更粗粒度的方式，一下是常见的几种精度的记忆集。

    1. 字长精度：每条记录精确到一个机器字长（处理器的寻址地址，如常见的32位和64位），该字内包含跨代指针。
    2. 对象精度：每条记录精确到一个对象，该对象内包含跨代指针。
    3. 卡精度：每条记录精确到一块内存区域，该区域内包含跨代指针。

  3. 上面的卡精度的记忆集就是卡表（Card Table），也是最常见的记忆集的实现方式，这里也需要注意，记忆集是JVM的规范，并未规定其具体实现，而卡表则是记忆集的一种具体实现了。卡表的最简单形式便是一个字节数组，而在HotSpot中也是这么实现的。字节数组CARD_TABLE中的每一个元素对应的都是一块内存区域，这块内存区域叫做卡页（Card Page），HotSpot中卡页大小为512字节。一个卡页中通常包含不止一个对象，如果该卡页中的对象有跨代指针，那么就将该区域对应的数组元素的值变为1，称为变脏了（dirty），否则为0.

3.2.5 写屏障

  1. 既然需要使用卡表来帮助缩小GC Roots的扫描范围，那么怎么维护卡表便是一个重要的问题。理论上来说，每当有其他区域的对象引用了本区域的对象时，卡表中对应区域就应该变脏，对于解释执行的字节码来说，JVM负责字节码指令的执行，所以JVM有充分的介入空间；但是如果是编译执行呢？即时编译后的代码已经是机器指令流了，JVM无法再介入。

  2. HotSpot中通过使用写屏障（Write Barrier）来维护卡表状态。写屏障就是JVM对“引用类型字段赋值”的AOP切面，在赋值前的屏障叫做写前屏障（Pre-Write Barrier），赋值后的屏障叫做写后屏障（Post-Write Barrier）。

  3. 一旦使用写屏障，JVM会为所有的赋值操作生成相应指令，收集器便会在引用类型赋值后更新卡表，虽然这样也会产生额外的开销，但是对比Minor GC时回收整个老年代要好多了。

  4. 在卡表的维护中，也会出现伪共享问题：

    1. 缓存行：现代CPU的缓存结构通常是以缓存行为单位进行操作的，缓存行的大小通常为64字节，当多个线程同时访问不同的变量时，如果这些变量位于同一缓存行中，那么每次更新缓存行中的某一个变量都会导致整个缓存行被重新加载，而这个过程需要从主存中获取缓存行的最新数据，这就会导致缓存一致性的开销和性能的下降。
    2. 由于一个卡表元素占一个字节，一个缓存行内有64个卡表元素，而64个卡表元素对应的卡页大小为32KB，如果说有多个线程要赋值的对象刚好处在这32KB的内存中，就会导致维护卡表时正好在同一个缓存行中影响性能。为了解决这个问题，可以先检查卡表标记，只有卡表元素不脏时才将其变脏。在 JDK 7 之后，HotSpot 虚拟机增加了一个新的参数-XX：+UseCondCardMark，用来决定是否开启卡表更新的条件判断。

3.2.6 并发的可达性分析

  1. 为什么需要在一个可以保障一致性的快照中才可以进行从GC Roots的对象图遍历？

    白色：尚未被垃圾收集器访问过的对象。如果在标记阶段结束后仍是白色，则代表对象不可达。

    黑色：被垃圾收集器访问过，且该对象的所有引用都被扫描过。

    灰色：被垃圾收集器访问过，但该对象的至少一个引用没有被扫描。

    如果用户线程和垃圾回收线程并发执行，那么可能出现两个问题：

    ​ 1.用户线程在收集器将对象标记为可达后切断了引用关系。勉强可以接受，只是多了一点垃圾而已；

    ​ 2.用户线程在收集器将对象标记为不可达后建立了引用关系。不可以接受，程序肯定会出现错误；

  2. 只有当一下两个条件同时满足时，才会发生对象消失的问题：

    1. 插入了一条或多条从黑色对象到白色对象的新引用；
    2. 删除了所有从灰色对象到白色对象的直接、间接引用。

  3. 所以为了消除这个问题，只需要破坏上面两个条件之一即可，故有两种解决方案：增量更新、原始快照。

  4. 增量更新：破坏第一个条件。当插入黑色对象指向白色对象的新引用时，将这个新插入的引用记录，在扫描结束后将记录的引用关系中的黑色对象作为根，重新扫描；其实也就是将插入新引用的黑色对象变为灰色对象。

  5. 原始快照：破坏第二个条件。当灰色对象需要删除指向白色对象的引用时，需要将删除的引用记录，在扫描结束后，以记录的灰色对象为根，重新扫描。无论引用关系删除与否，都会按照刚刚开始的那一刻的对象快照来搜索。

3.3 经典垃圾收集器

3.3.1 Serial收集器（新生代）

  1. 采用标记复制算法、单线程工作的垃圾收集器（在垃圾收集时，必须暂停其他的工作线程）。

  2. 依然是客户端模式下默认的新生代收集器，对于内存资源受限的环境，是所有收集器中额外内存消耗最小的（为保证垃圾收集顺利进行而额外存储的信息），且对于核心数较少的处理器没有线程交互的开销。

3.3.2 ParNew收集器（新生代）

  1. Serial收集器的多线程版本。

3.3.3 Parallel Scavenge收集器（新生代）

  1. 同样是针对新生代，同样是采用标记-复制算法，Parallel Scavenge的目标是达到一个可以控制的吞吐量。

    吞吐量=运行用户代码时间运行用户代码时间+运行垃圾收集时间吞吐量=\frac{运行用户代码时间运行}{用户代码时间+运行垃圾收集时间} 吞吐量=用户代码时间+运行垃圾收集时间运行用户代码时间运行​

  2. Parallel Scavenge 收集器提供了两个参数用于精确控制吞吐量，分别是控制最大垃圾收集停顿时间的-XX：MaxGCPauseMillis 参数以及直接设置垃圾收集时间站总时间比例的-XX：GCTimeRatio 参数。

  3. Parallel Scavenge 收集器还有一个参数-XX： +UseAdaptiveSizePolicy 值得我们关注。这是一个开关参数，当这个参数被激活之后，就不需要人工指定新生代的大小（-Xmn）、Eden 与 Survivor 区的比例（-XX：SurvivorRatio）、晋升老年代对象大小（-XX：PretenureSizeThreshold）等细节参数了，虚拟机会根据当前系统的运行情况收集性能监控信息，动态调整这些参数以提供最合适的停顿时间或者最大的吞吐量。这种调节方式称为垃圾收集的自适应的调节策略（GC Ergonomics）。如果对于收集器运作不太了解，手工优化存在困难的话，使用Parallel Scavenge 收集器配合自适应调节策略，把内存管理的调优任务交给虚拟机去完成也许是一个很不错的选择。只需要把基本的内存数据设置好（如-Xmx 设置最大堆），然后使用-XX：MaxGCPauseMillis 参数（更关注最大停顿时间）或-XX：GCTimeRatio （更关注吞吐量）参数给虚拟机设立一个优化目标，那具体细节参数的调节工作就由虚拟机完成了。自适应调节策略也是 Parallel Scavenge 收集器区别于 ParNew 收集器的一个重要特性。

3.3.4 Serial Old收集器（老年代）

  1. Serial收集器的老年代版本，同样是单线程的，使用标记整理算法。

3.3.5 Parallel Old收集器（老年代）

  1. Parallel Scavenge的老年代版本，多线程并发收集，基于标记整理算法。

3.3.6 CMS（Concurrent Mark Sweep）收集器

  1. 基于标记-清除算法。总共有四个阶段：初始标记（Cms initial mark），并发标记（CMS concurrent mark），重新标记（CMS remark），并发清除（CMS concurrent sweep）。

  2. 初始标记、重新标记依然需要STT。初始标记仅仅是根节点枚举，速度很快；并发标记是从GC Roots开始遍历整个对象图的过程，此过程很长但是可以和用户线程一起并发执行；重新标记则是采用了增量更新的方式来解决对象消失的问题；最后是并发清除，由于不需要移动对象，所以可以和用户线程并发执行。

  3. CMS默认的回收线程数是（处理器核心线程数+3）/4，如果处理器的核心线程数在4个及以上时，垃圾回收线程所占用的处理器资源不超过25%；但如果是双核或单核处理器时，就需要占用大量的运算能力用以垃圾回收，用户程序的执行速度可能会大幅降低。

  4. CMS也无法处理浮动垃圾，有可能导致“Con-current Mode Failure”失败从而导致一次STT的Full GC。在并发标记和并发清除阶段，用户线程和垃圾回收线程是交替执行的，所以在此过程中也会产生垃圾，但是此时CMS无法在此次的垃圾收集中将它们回收，只能等到下次，这部分的垃圾就称为浮动垃圾。并且因为垃圾回收线程和用户线程的并发执行，需要在回收的过程中预留一部分的内存空间给用户线程使用，所以CMS无法和其他垃圾回收器一样可以等到老年代快填满了在进行垃圾回收。JDK5中CMS默认当老年代使用了68%的空间时就会进行垃圾回收，该参数可以通过-XX:CMSInitiatingOccupancyFraction的值来调整；JDK6时的CMS启动阈值调整为了92%。如果在CMS运行期间预留的内存无法满足用户线程的需要，那么就会出现一次并发失败（Concurrent Mode Failure），此时JVM就不得不临时启用Serial Old重新整理老年代。

  5. 同时，CMS也是一款基于标记清除算法的垃圾收集器，所以会出现内存空间碎片化严重从而无法分配大对象的问题，个-XX： +UseCMSCompactAtFullCollection 开关参数（默认是开启的，此参数从 JDK 9 开始废弃），用于在 CMS 收集器不得不进行 Full GC 时开启内存碎片的合并整理过程，由于这个内存整理必须移动存活对象，（在 Shenandoah 和 ZGC 出现前）是无法并发的。然而这样停顿的时间又会延长，所以参数-XX：CMSFullGCsBeforeCompaction（此参数从 JDK 9 开始废弃），这个参数的作用是要求 CMS 收集器在执行过若干次（数量由参数值决定）不整理空间的 Full GC 之后，下一次进入 Full GC 前会先进行碎片整理（默认值为 0，表示每次进入 Full GC 时都进行碎片整理）。

3.3.7 G1（Garbage First）收集器

  1. G1的初衷是希望能够建立出“停顿时间模型”的垃圾收集器，即能够支持在指定的M毫秒的时间片段内，消耗在垃圾收集上的时间大概率不超过N毫秒的目标。为了实现该目标，G1提出了一种新的基于Region的堆内存布局：G1可以面向任何部分组成回收集（Collection Set）进行回收，而划分的标准也不再是分代，而是根据哪一块内存的垃圾最多，收益最大。这就是G1的Mixed GC模式。

  2. G1的内存布局如下：

    1. G1也遵循分代收集理论，但是不再坚持固定大小及固定数量的分代区域的划分，而是把Java堆划分成多个大小等大的Region，每一个Region都可以是Eden、Survivor、老年代，收集器针对不同的Region采取不同策略。

    2. Region中还有一种特殊的Humongous区域，用以存储大对象。G1认为只要一个对象的大小超过了一个Region大小的一半即可认定为大对象，Region的大小可以通过-XX：G1HeapRegionSize设定，取值为1MB-32MB之间的任意2的N次幂数。如果对象的大小超过了一个Region的大小，则会存放在多个连续的Region中，G1中的大多数行为都会将Humongous当作老年代的一部分看待。

    3. G1通过将Region作为单次回收的最小单元，有计划的避免在整个Java堆中进行全区域的垃圾收集的方式，能够建立可预测的停顿时间模型。具体做法就是：是让 G1 收集器去跟踪各个 Region 里面的垃圾堆积的“价值”大小，价值即回收所获得的空间大小以及回收所需时间的经验值，然后在后台维护一个优先级列表，每次根据用户设定允许的收集停顿时间（使用参数-XX：MaxGCPauseMillis指定，默认值是 200 毫秒），优先处理回收价值收益最大的那些 Region，这也就是“Garbage First”名字的由来。

  3. G1的一些问题

    1. 如何解决跨Region引用问题？可以使用记忆集来避免全堆的扫描。但是在G1中，每个Region都需要维护自己的记忆集，这种记忆集在CardTable基础上增加了哈希表，Key是其他Region的起始地址，Value是一个卡表索引号的集合，也就是说每个Region的记忆集元素数量都有Region总数-1个，key代表其他的Region，value代表该Region中的哪些卡页指向了自己Region中的对象；同时，如果该CardTable中对应元素的卡页指向了其他Region的对象，那么对应元素的值为1（变脏了）。这种记忆集比之前的记忆集都复杂，并且Region的数量明显大于分代数量，所以内存占用比较大。通常来说，G1至少需要Java堆容量10%-20%的额外内存来维持工作

    2. 如何解决并发标记过程中用户线程并发执行导致错误的问题？G1使用的是原始快照的方式。那么如何解决垃圾回收过程中新对象的创建问题呢？G1为每个Region设计了两个TAMS(Top at Mark Start)的指针，分别为previous TAMS和next TAMS，分别代表上一轮并发标记中隐式存活的top指针和这一轮存活的top指针。

      bottom---------previous TAMS----------next TAMS----------top

      previous TAMS—next TAMS之间的对象就是上一次隐式标记存活的对象，next TAMS—top之间的对象就是此次并发标记隐式标记存活的对象。

      同时，G1也有CMS的毛病：如果内存回收的速度赶不上内存分配的速度，那么G1也会冻结用户线程执行FULL GC导致长时间的STT。

    3. G1如何建立起可靠的停顿预测模型呢？G1的停顿预测模型是以衰减均值为理论基础实现的，所谓的衰减均值就是更容易受到新数据印象的平均值。G1会记录每个Region的回收耗时、每个Region记忆集的脏卡数量等每一个可以测量的步骤花费的成本，并分析得出平均值、标准偏差、置信度等统计信息，通过这些信息来预测由哪些Region组成回收集才不会超过期望停顿时间。

  4. G1的运行过程：

    1. 初始标记：标记GC Roots，修改TAMS指针，让下一阶段用户线程并发时，可以正确的在可用Region中分配对象。借用进行Minor GC的时候同步完成？

    2. 并发标记：开始进行可达性分析，并且在对象图扫描完成后，处理原始快照（SATB）记录的在并发时引用变动的对象。

    3. 最终标记：短暂的STT，处理SATB记录。

    4. 筛选回收：更新Region的统计数据，对各个Region的回收价值和成本进行排序，根据用户期望停顿时间选择任意多个Region组成回收集，将回收集Region中的存活对象复制到空的Region中，将旧Region直接回收。因为这一部分需要移动对象，必须STT，由多条垃圾回收线程并行完成。
