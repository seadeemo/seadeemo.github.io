3.schema的设计

SCHEMA的设计

1 数据类型

1.1 整数类型

分别有TINYINT、SMALLINT、MEDIUMINT、INT、BIGINT，分别使用8、16、24、32、64位的存储空间，存储值的范围是

−2(N−1)到2(N−1)−1-2^{(N-1)} 到 2^{(N-1)}-1 −2(N−1)到2(N−1)−1

对于类型的选择，这些只会影响MySql对于这些数据在磁盘和内存中的保存方式，整数的计算通常都会使用BIGINT来计算（一些聚合函数除外）。

整数类型有可选的UNSIGNED属性，表示不允许为负数。同时，MySql的整数类型也可以指定宽度，但是通常来说没有意义，这并不会限制值的合法范围，只是规定了交互工具上显示字符的个数，存储、计算的消耗是没有变化的。

1.2 实数类型

分别有浮点类型的FLOAT、DOUBLE以及可以精确计算的定点类型的DECIMAL，FLOAT和DOUBLE类型分别占用4字节和8字节的空间，同时对于浮点类型的计算，MySql会使用DOUBLE类型。

DECIMAL可以通过一个或两个参数来指定精度及范围，如DECIMAL(p,s)指总位数为p位，小数位为s位，默认情况下p为10，s为0、DECIMAL§指总位数为p位，小数位为0 。

对于Decimal，MySQL会将数字打包保存进一个二进制字符串中(每4个字节存9个数字)，比如Decimal(18,9)就需要使用9个字节，小数点左边的9个数字占4字节，右边的9个数字占4字节，小数点占一字节。

由于额外的空间成本和计算成本问题，应该尽量只在需要精确计算小数的场景下使用DECIMAL，例如财务管理系统；但是如果数据量很大，可以考虑使用BIGINT来代替：将需要存储的数据乘以一定的系数从而变成一个最小整数存入MySql中。

1.3 字符串类型

  1. VARCHAR、CHAR

    因为VARCHAR和CHAR的存储和存储引擎有关系，所以在这里假定使用的是InnoDB引擎。

    1. VARCHAR是长度可变的字符串，所以需要额外的1或2字节用以存储字符串的长度：如果列的最大长度<=255字节，则需要一个字节存储字符串长度；否则使用两个字节。假设使用latin1字符集，一个VARCHAR(10)的列需要11字节的存储空间，而VARCHAR(1000)的列需要1002字节的存储空间。

      由于VARCHAR的长度可变，故更新时有可能会变化，这会导致额外的工作。如果行的增长使得原来位置无法容纳更多的内容，不同处理引擎会采取不同的操作来处理，InnoDB会分裂页使行可以放进页内。

      以下场景适合使用VARCHAR：字符串列的最大长度远大于平均长度；列的更新比较少；使用了UTF-8这样的字符集，每个字符使用不同的字节数来存储。

    2. CHAR是固定长度的字符串，适合存储非常短的字符串，或者值的长度基本相同的字符串。在存储CHAR值时，MySQL会删除所有的末尾空格，而在比较时MySQL又会根据需要填充空格。

    3. 如果需要一列存储true\false，可以使用CHAR(0)来存储，该列可以允许null和空字符串。

  2. BLOB和TEXT

    1. BLOB和TEXT都是为了存储很大数据而设计的字符串数据类型，分别采用二进制和字符方式存储。
    2. TEXT类型有TINYTEXT、SMALLTEXT、TEXT、MEDIUMTEXT、LONGTEXT，BLOB有TINYBLOB、SMALLBLOB、BLOB、MEDIUMBLOB、LONGBLOB。
    3. MySQL会将TEXT和BLOB当作独立的对象处理，存储引擎在存储时通常会做特殊处理。当BLOB和TEXT的值太大时，InnoDB会使用专门的“外部”存储区域来进行存储，此时每个值需要1~4个字节的空间存储一个指针，在外部区域存储实际的值。
    4. MySQL不能将BLOB和TEXT数据类型的完整字符串放入索引，也不能使用索引进行排序。

  3. 枚举(ENUM)

    1. 枚举列可以把一些不重复的字符存成一个预定义的集合，MySQL在存储枚举时非常紧凑，会根据列表值的数量压缩到一个或两个字节中。

      需要注意，即使给枚举列设定了默认值，当新增时传入的值不在枚举值范围内时，会报错。

    3. 对于枚举来说，由于存储时是将每个枚举值存储为整数，所以查找时需要将整数转换为字符串后才可以。对于存储时的开销，枚举更小；对于枚举和枚举的关联查询，比varchar和varchar的关联查询更快，但是枚举和varchar的关联查询则更慢。

1.4 日期和时间类型

  1. DATETIME

    可以保存大范围的值，从1001年到9999年，精度为秒。它将日期封装进格式为YYYYMMDDHHMMSS的整数当中，与时区无关，使用8字节的存储空间。

  2. TIMESTAMP

    保存了从1970年1月1日物业以来的秒数，只使用4字节的存储空间，只能表示从1970年到2038年。TIMESTAMP显示的值依赖于时区，而DATETIME保留文本表示的日期和时间。

    TIMESTAMP默认为NOT NULL。

  3. 通常情况下应该尽量使用TIMESTAMP，因为比DATETIME的空间效率更高，如果需要存储更小粒度的时间或者更大范围的时间怎么办？可以使用BIGINT存储微秒级别的时间戳，也可以使用DOUBLE存储秒之后的小数部位。

  4. 关于时区的问题，目前主流的处理方式是将日期时间存储为时间戳，并以协调世界时(UTC)的形式，避免MySQL处理的复杂性。

1.5 位数据类型

  1. BIT

    BIT(1)定义一个包含1个位的字段，BIT(2)定义包含2个位的字段，BIT列的最大长度为64位。也就是说，定义一个BIT(4)的字段，可以在这一列中存储4个true\false的值。

    然而MySQL会把BIT当作字符串类型，而不是数字类型，这是很奇怪的。例如存储一个b ‘00111001’的值（二进制值位57）到BIT(8)的列内并检索他，得到的结果是’9’（ASCII码为57的字符），但是如果检索b+0得到的就是57（因为是数字上下文）。

    如果需要列保存单个true\false的值，可以使用CHAR(0)；如果需要保存多个true\false的值，可以使用SET，

  2. SET

    如果需要保存很多的true/false值，可以将这些列合并为一个SET数据类型。

  3. 整数列上的位操作

    可以使用一个TINYINT来替代一个8个元素的SET，为每个位定义名称常量即可。

1.6 JSON

如果将某张表的所有字段通过JSON的方式来存储，可能会牺牲部分性能：存储空间上来说，一条包含n个字段的JSON数据比一条包含n个字段的SQL行数据所占空间大；查询时间上来说，一条包含n个字段的JSON数据比一条包含n个字段的SQL行数据查询时间多。虽然他们之间的差距比较小，但是在高并发的情况下也会积少成多。

2 选择标识列

对于标识列来说，数据类型的选择有以下几个注意点：

  1. 尽量使用整数类型，因为速度很快并且可以AUTO_INCREAMENT
  2. 避免使用字符串类型，不仅是因为消耗空间大，也是因为他们的速度慢。尤其是对于UUID、MD5等类型的字符串，尽量将其变为整数类型后再存入MySQL。

3 MySQL中schema设计的缺陷

3.1 过多的列

MySQL的存储引擎API工作时需要在服务器层和存储引擎层之间通过行缓冲格式来拷贝数据，之后由服务器层将行缓冲数据解码成各个列，这个操作的代价相当高，InnoDB的行结构总是需要转换。而转换的代价依赖于列的数量，也就是说，一张表中列的数量过多时，会导致CPU的占用相当高。

3.2 过多的关联

对于MySQL来说，每个关联操作最多只能关联61张表，即使关联的表数小于61，过多的关联也会加大解析和优化查询的代价。最好将关联查询控制在12张表以内。

3.3 枚举

尽量使用字段来关联到字典表而不是用枚举。在5.0之前ALTER TABLE是一种阻塞操作；即使在5.1后如果不是在列表的最后增加值也是需要ALTER TABLE的。

4 范式

4.1 什么是数据库三范式

  1. 第一范式：符合1NF的关系中的每个属性都不可再分
  2. 第二范式：2NF在1NF的基础之上，消除了非主属性（包含在码中的属性为主属性）对于码（除码以外的属性都完全依赖于码）的部分函数依赖。
  3. 3NF在2NF的基础之上，消除了非主属性对于码的传递函数依赖。

4.2 范式的优点

  1. 范式化的更新操作通常比反范式化要快
  2. 当数据较好的范式化时，就只有很少或没有重复数据，执行操作会更快
  3. 范式化的表通常更小，可以更好的放入内存中，执行操作更快
  4. 很少有重复数据意味着检索列表数据时更少需要DISTINCT或GROUP BY

4.3 反范式的优点

可以更好的避免关联查询，提高性能。

1.MySQL的架构

1 MySQL的逻辑架构

  1. 第一层包括连接处理、身份验证、确保安全性等。
  2. 大多数MySQL的核心功能都在这一层，包括查询解析、分析、优化、以及所有的内置函数（例如，日期、时间、数学和加密函数），所有跨存储引擎的功能也都在这一层实现：存储过程、触发器、视图等。
  3. 第三层是存储引擎层。存储引擎负责MySQL中数据的存储和提取。服务器通过存储引擎API进行通信。这些API屏蔽了不同存储引擎之间的差异，使得它们对上面的查询层基本上是透明的。存储引擎层还包含几十个底层函数，用于执行诸如“开始一个事务”或者“根据主键提取一行记录”等操作。但存储引擎不会去解析SQL，不同存储引擎之间也不会相互通信，而只是简单地响应服务器的请求。

1.1 连接的管理和安全性

  1. 默认情况下，每个客户端连接都会在服务器进程中拥有一个线程，该连接的查询只会在这个单独的线程中执行，该线程驻留在一个内核或者CPU上。服务器维护了一个缓存区，用于存放已就绪的线程，因此不需要为每个新的连接创建或者销毁线程。
  2. 当客户端（应用）连接到MySQL服务器时，服务器需要对其进行身份验证。身份验证基于用户名、发起的主机名和密码。客户端连接成功后，服务器会继续验证该客户端是否具有其发出的每个查询的权限.

1.2 优化与执行

  1. MySQL解析查询后会创建解析树，并对齐进行各种优化，如重写查询、决定表的读取顺序、选择合适的索引等。
  2. 优化器并不关心使用的是什么存储引擎，但是因为存储引擎堆查询优化的结果有一定影响，所以优化器会询问存储引擎的一些功能或操作的成本，以及表数据的统计信息，从而更好的优化。
  3. 旧版本中MySQL有内部查询缓存可以直接返回查询结果，但是从5.7.20开始，查询缓存被标注为弃用的特性，在8.0中完全移除。

2 并发控制

从服务器级别和存储引擎级别，简要的讨论下MySQL的并发控制。

2.1 读写锁

  1. 通常来说，处理并发读/写的系统会实现一个有两种锁类型组成的锁系统，分别为共享锁和排他锁，或者是读锁和写锁。
  2. 读锁与读锁之间相互不阻塞，而写锁会阻塞写锁和读锁。

2.2 锁的粒度

  1. 表锁：写操作时锁住整张表，阻塞所有其他的读写操作。开销最小。
  2. 行锁：写操作时锁住当前行，不会阻塞其他行的读写操作。最大程度支持并发操作，但是锁开销也很大。行锁的实现是由存储引擎完成的，服务器通常都不知道存储引擎中锁的实现方式。

3 事务

  1. ACID（Atomicity,Consistency,Isolation,Durability）
    1. Atomicity（原子性）：一个事务中的操作要么全部成功，要么全部失败。
    2. Consistency（一致性）：数据库总是从一个一致性状态切换到下一个一致性状态，也就是说一个事务中所作的任何修改，如果没有最终提交，是不会保存在数据库中的，数据库的变化是以事务为单位的。
    3. Isolation（隔离性）：一个事务所作的修改在最终提交前对其他事务是不可见的。
    4. Durability（持久性）：一旦提交，事务所作的修改就被永久的保存至数据库中。
  2. 隔离级别
    1. READ UNCOMMITED（读未提交）：在事务中可以查看其他事务未提交的数据。
    2. READ COMMITTED（读已提交）：一个事务可以看到其他事务在它开始之后提交的修改，但是仍然会有不可重复读的问题，即一个事务对于同一行数据查看两次可能会有不同的结果（其他事务修改并提交了）。
    3. REPEATABLE READ（可重复读）：保证了在同一个事务中多次读取相同行数据的结果是一样的，但是仍然会有幻读问题，即事务在多次读取范围内的数据时，另一个事务在该范围内插入了一条新数据并提交，那么此事务会读出幻行。InnoDB和XtraDB存储引擎通过多版本并发控制（MVCC）解决了幻读问题。
    4. SERIALIZABLE（串行化）：即所有事务按序执行。
  3. 死锁。一旦发生死锁，不回滚其中的一个事务就无法打破死锁，在InnoDB中，会将持有最少行级排他锁的事务回滚。
  4. 事务日志：事务日志有助于提高事务的效率，存储引擎只需要更改内存中的数据副本，然后将更改的记录写入事务日志中，再有某一个后台进程在某个时间根据事务日志去更新表。因为事务日志存放在硬盘中一块连续的存储空间上，并且采用的是追加写的操作，所以写入事务日志比直接修改磁盘中的表要快得多。
  5. 隐式锁定和显示锁定：隐式锁定即MySQL自行随事务获取和释放锁，而显示锁定可以通过Sql语句来实现：select … for share（MySQL8.0新增，用于取代select … lock in share mode） 或者 select … for update。

4 多版本并发控制(MVCC)

  1. 5.7 InnoDB的MVCC

    1. 是通过每行记录后面的两个隐藏列完成的，一个保存了行的创建版本号，一个保存了行的过期版本号（删除版本号），每开始一个新的事务，系统版本号都会自动递增，事务开始时刻的系统版本号会作为事务的版本号，用来和查询到的版本号进行比较。

    2. Select：

      1. InnoDB只会查找版本号早于或等于当前事务版本号的数据行，这样可以保证该数据要么是事务开始之前就已经存在，或者是由当前事务修改或添加的。
      2. 行的删除版本号要么不存在，要么大于当前版本号，保证数据在事务开始之前是没有被删除的。

      只有满足这两个条件，InnoDB才会将其作为查询结果返回。

    3. Insert

      InnoDB为新插入的行保存当前系统版本号作为行版本号

    4. Delete

      InnoDB为删除的每一行保存当前系统版本号作为行的删除标识

    5. Update

      InnoDB插入一行新纪录，保存当前系统版本号作为行版本号，同时保存当前系统版本号到原来的行作为行删除标识。

  2. 8.0 InnoDB的MVCC

    1. 首先，每个事务将会在首次读取任意数据时被分配一个事务ID，当在事务中修改数据时，会向Undo日志中写入一条如何恢复该更改的Undo记录，并且该事务的回滚指针将会指向该条Undo日志记录，如果有多个对话读取聚簇主键索引记录（实际上就是InnoDB中的一张表中的数据）时，InnoDB会将该数据的事务ID和会话的读取视图（读取记录实际上就是事务开始之前的快照，记录在此事务开始前已经提交的事务ID以及版本号等信息）进行比较，如果当前状态下的数据不应被该事务可见的话（此数据的事务还未提交），那么该事务会使用Undo日志中的记录并跟踪这条记录（这样这个事务就对该数据不可见），直到会话达到一个符合可见条件的事务ID上。

    2. 所有的Undo日志写入也会写入到Redo日志中，因为Undo日志的写入也是服务器崩溃后恢复过程的一部分（因为需要Undo日志来做MVCC），且Undo日志是事务性的（Redo日志中记录了每个事务对数据库的修改操作，用以服务器崩溃后的数据恢复）。

    3. 需要注意，Undo日志也是MySQL8.0新增的特性

  3. 需要注意，MVCC和REPEATABLE READ以及READ COMMITED两个隔离级别兼容，并不适用于SERIALIZABLE和READ UNCOMMITTED两个隔离级别。

5 复制

MySQL被设计为在任意时间点上集群中只有一个节点上接受写操作，如果需要写入多台服务器中，就需要使用MySQL原生的支持方式：复制。所谓复制，就是MySQL源节点为每一个副本节点提供一个线程，该线程用作复制客户端登陆，当源节点写入发生时，该线程被唤醒，向副本节点发送数据。

6 数据文件结构

在8.0版本中，MySQL将表的元数据重新设计为一种数据字典，包含在表的.ibd文件中。这使得表结构上的信息支持事务和原子级数据定义更改。在操作期间，我们不再仅仅依赖information_schema来检索表定义和元数据，而是引入了字典对象缓存，这是一种基于最近最少使用（LRU）的内存缓存，包括分区定义、表定义、存储程序定义、字符集和排序信息。服务器访问表的元数据的方式的这一重大变化减少了I/O，非常高效。特别是当前访问最活跃的那些表，在缓存中最常出现。每个表的.ibd和.frm文件被替换为已经被序列化的字典信息 （.sdi）。

2.虚拟机执行子系统

虚拟机执行子系统

1 类文件结构

1.1 Class文件格式

  1. 任何一个Class文件对应的是一个唯一的类或者接口的定义信息，但是类或者接口并不一定非要定义在文件中（可以通过动态生成直接送入类加载器中），所谓的Class文件格式只是任何一个类或者接口应当满足的格式。

  2. Class文件是一组以8个字节为基础单位的二进制流，各个数据项目严格按照顺序排列在文件之中，中间没有分隔符，所以其中存储的内容几乎全部是程序运行的必要数据。如果有占用8字节以上的数据项时，会按照高位在前（Big-Endian，高位字节存储在在地址最低位，而低位字节在地址最高位）的方式分割成若干个8字节进行存储。

  3. Class文件是16进制而非2进制的。

  4. Class文件格式采用一种类似C语言中结构体的伪结构来存储数据，该伪结构中只有两种数据类型：无符号数、表。

    1. 无符号数：基本的数据类型，以u1、u2、u4、u8来分别表示1、2、4、8个字节的无符号数，无符号数可以用来描述数字、索引引用、数量值或者UTF-8编码的字符串值。

    2. 表：是由多个无符号数或其他表作为数据项构成的复合数据类型，为便于区分，所有表命名都习惯以_info结尾。整个Class文件本质上也可以看作一张表。

  5. 无论是无符号数还是表，当需要描述同一类型但数量不定的多个数据时，经常会使用一个前置的容量计数器加若干个连续的数据项的形式，这时候称这一系列连续的某一类型的数据为某一类型的“集合”。

1.1.1 魔数和Class文件的版本

  1. 每个Class文件的头4个字节被称为魔数，唯一作用是确定该文件是否是能被虚拟机接受的Class文件，Class文件的魔数是0xCAFEBABE。
  2. 在魔数之后的4个字节为Class文件的版本号，第5、6个字节是次版本号，第7、8个字节是主版本号。

1.1.2 常量池

  1. 紧接着主次版本号之后的是常量池入口。由于常量池中的常量数量不是固定的，所以常量池的入口才需要放置一项u2类型的数据，代表常量池容量计数值(constant_pool_count)，此容量计数从1开始。例如如果容量为0x0016，即十六进制的16 十进制的22，那么总共有21项常量，索引范围为1~21；而特地将第0项常量空出来有特殊考虑，如果某些指向常量池中的数据的索引需要表示“不引用任何一个常量池项目”的含义，可以指向第0个索引。

  1. 常量池中主要存放两大类常量：字面量（Literal）和符号引用（Symbolic References）。

    1. 字面量：

      类似Java的常量概念，如文本字符串、被声明为final的常量值等。

    2. 符号引用：

      因为Class文件不会保存各个方法、字段最终在内存中的布局，JVM只会在运行时或类创建时从常量池中获取对应的符号引用，之后再解析、翻译得到具体的内存地址。

      符号引用主要包括以下几类常量：

      1. 被模块到处或开放的包（Package）
      2. 类和接口的全限定名（Fully Qualified Name）
      3. 字段的名称和描述符（Descriptor）
      4. 方法的名称和描述符
      5. 方法句柄和方法类型（Method Handle、Method Type、Invoke Dynamic）
      6. 动态调用点和动态常量（Dynamically-Computed Call Site、Dynamically-Computed Constant）

    3. 常量池中每一项常量都是一个表，截至JDK13，常量表中分别由17中不同类型的常量。

AQS(AbstractedQueuedSynchronizer)

1 AQS概述

AQS(AbstractQueuedSynchronized)的核心思想是基于CLH队列锁的。

CLH队列锁即一个虚拟队列(只有结点，没有队列的实现)，包含多个结点和一个指向最后一个结点的尾指针tail。其中每个结点有两个重要属性：该结点所代表的线程的指针、该结点是否持有锁的状态变量。每当一个线程想要获取锁时，会先对tail结点进行getAndSet的原子操作，返回当前tail节点并使tail指向新入队的结点。入队后的结点会轮询上一个结点的状态，当上一个结点释放锁后会得到锁。注意，CLH并没有在每一个结点中显示的维护上一个结点的指针，仅仅是轮询还是那个个结点的状态。

实例:

  1. 定时计算的排行榜

    使用lrange分页查看list中的数据,适用于定时计算的排行榜存入list中,每次查询时直接查询list中数据即可.但是并不适用于即时计算的排行榜

  2. 最新列表

    lpush和lrange都可以实现最新列表的功能,但是当列表更新频繁时,有可能导致元素重复或者遗漏.当前列表里由表头到表尾依次有（E，D，C，B，A）五个元素，每页获取3个元素，用户第一次获取到（E，D，C）三个元素，然后表头新增了一个元素F，列表变成了（F，E，D，C，B，A），此时用户取第二页拿到（C，B，A），元素C重复了。只有不需要分页（比如每次都只取列表的前5个元素）或者更新频率低（比如每天凌晨更新一次）的列表才适合用list类型实现。

2.4 Set

自动去重的String类型的无序集合，底层是一个hash表（与Java类似）

  1. SADD key member [member …] 将一个或多个member元素加入到集合 key 中，已经存在的 member 元素将被忽略

  2. SMEMBERS key 取出该集合的所有值。(set中元素很多时应该谨慎使用,资源开销很大)

  3. SSCAN key cursor [MATCH pattern] [COUNT count]

    • cursor 参数。SSCAN 命令每次被调用之后， 都会向用户返回一个新的游标， 用户在下次迭代时需要使用这个新游标作为 SSCAN 命令的游标参数， 以此来延续之前的迭代过程。
      当 SCAN 命令的游标参数被设置为 0 时， 服务器将开始一次新的迭代， 而当服务器向用户返回值为 0 的游标时， 表示迭代已结束
    • count 参数。虽然增量式迭代命令不保证每次迭代所返回的元素数量， 但我们可以使用 COUNT 选项， 对命令的行为进行一定程度上的调整。 基本上， COUNT 选项的作用就是让用户告知迭代命令， 在每次迭代中应该从数据集里返回多少元素.也就是说,redis会返回count数量附近的元素,不一定返回的元素数量就是count
    • MATCH 参数。 和 KEYS 命令一样， 增量式迭代命令也可以通过提供一个 glob 风格的模式参数， 让命令只返回和给定模式相匹配的元素， 这一点可以通过在执行增量式迭代命令时， 通过给定 MATCH pattern 参数来实现

  4. SISMEMBER key member 判断集合<key>是否为含有该值，有1，没有0

  5. SCARD key 返回该集合的元素个数。

  6. SREM key member [member …] 删除集合中的某个元素。

  7. SPOP key [count] 随机从该集合中吐出值。

  8. SRANDMEMBER key [count] 随机从该集合中取出n个值。不会从集合中删除 。

  9. SMOVE source destination member 把集合中一个值从一个集合移动到另一个集合

  10. SINTER key [key …] 返回多个集合的交集元素。

  11. SUNION key [key …] 返回两个集合的并集元素。

  12. SDIFF key [key …] 返回两个集合的差集元素(key1中的，不包含key2中的)

  13. SUNIONSTORE destination key [key …] 将多个key的并集存在destination中

  14. SINTERSTORE destination key [key …] 将多个key的交集存在destination中

  15. SDIFFSTORE destination key [key …] 将多个key的差集存在destination中

3.10.1 ACL

ACL 即 Access Control List—访问控制列表,实际上是一种给特定用户添加特定权限的表达式

需要注意,ACL使从左向右解析的

user <username> …acl rules…

ACLrules 意义
on 启用该用户
off 禁用该用户,但注意:已使用该用户的连接仍然可以使用
skip-sanitize-payload RESTORE dump-payload sanitation is skipped.
sanitize-payload RESTORE dump-payload is sanitized (default).
+<command> 允许该命令的执行
-<command> 不允许该命令的执行
+@<category> 允许某一类命令的执行,例如+@all,+@set,+@admin
+<command>|subcommand Allow a specific subcommand of an otherwise disabled command. Note that this form is not allowed as negative like -DEBUG only additive starting with “+”.
allcommands 和+@all同理
nocommands 和-@all同理
~<pattern> Add a pattern of keys that can be mentioned as part of commands. For instance ~* allows all the keys. The pattern is a glob-style pattern like the one of KEYS. It is possible to specify multiple patterns.
allkeys 和~*同理
resetkeys 刷新允许的key的模板
&<pattern> Add a glob-style pattern of Pub/Sub channels that can be accessed by the user. It is possible to specify multiple channel patterns.
allchannels 和&*同理
resetchannels Flush the list of allowed channel patterns.
><password> 向用户添加指定的可用的密码
<<password> 从该用户可用密码中移除某指定密码
nopass 该用户所有的密码被移除,且无需密码.若使用在用户default上,则会使所有新连接拥有权限
resetpass 刷新可用密码列表
reset 执行以下所有命令 resetpass, resetkeys, off,-@all

3.12 Memory Management

3.12.1 maxmemory

设置最大内存占用大小,达到该值后使用淘汰策略减少内存占用

如果该redis是主节点,则需要留出一些系统内存空间给同步队列缓存,即小于系统内存大小(除非淘汰策略是noeviction)

3.12.2 maxmemory-policy

淘汰策略

volatile-lru -> Evict using approximated LRU, only keys with an expire set.
allkeys-lru -> Evict any key using approximated LRU.
volatile-lfu -> Evict using approximated LFU, only keys with an expire set.
allkeys-lfu -> Evict any key using approximated LFU.
volatile-random -> Remove a random key having an expire set.
allkeys-random -> Remove a random key, any key.
volatile-ttl -> Remove the key with the nearest expire time (minor TTL)
noeviction -> Don’t evict anything, just return an error on write operations.

LRU means Least Recently Used
LFU means Least Frequently Used

默认为: maxmemory-policy noeviction

3.12.3 maxmemory-samples

即当需要使用淘汰策略淘汰key时,总共选出n个key,然后选出最合适的一个进行删除.

值为5时比较合适,10则更加接近真实结果但耗费资源更多,3更快但不准确

maxmemory-samples 5

5.新数据类型

5.1 Bitmaps

数据结构如图所示

setbit<key><offset><value> 为指定key的指定下标设置0或1的值

getbit<key><offset> 获取指定key的指定下标的值

bitcount<key>[start end] 统计字符串从start字节到end字节比特值为1的数量

bitop and/or/not/xor <destkey> [key…] 多个Bitmaps的and（交集） 、 or（并集） 、 not（非） 、 xor（异或(同为0,异为1)） 操作并将结果保存在destkey中,比如:bitop and unique:users:and:20201104_03 unique:users:20201103 unique:users:20201104

实例:

每当用户访问某页面或某网站时,将某个bitmap中该用户id下标位置的值修改为1,代表该用户该日访问过.

可以通过getbit(获取某用户某日是否访问过),bitcount(某日多少用户访问过),bitop(其他操作)

需要注意:如果每天访问用户很少但用户基数很大的情况下,使用bitmap不如使用set节省空间

5.2 HyperLogLog

专门用以计算基数的数据类型.

比如,现在需要统计某网站的独立访客数量时(即一个用户访问多次只记一次),使用set,bitmaps,hash都可以实现,但是当数量到达百万千万级别时,则所需要的内存空间很大,此时就需要一种内存空间需求小的数据类型,此为HyperLogLog.

HyperLogLog的每个键只需要12KB的内存即可计算2^64的基数,当然有一定的误差,但是这个误差是被允许的.

pfadd <key><element> [element …] 添加指定元素到 HyperLogLog 中,成功返回1,不成功返回0

pfcount<key> [key …] 计算HLL的近似基数，可以计算多个HLL，比如用HLL存储每天的UV，计算一周的UV可以使用7天的UV合并计算即可

pfmerge<destkey><sourcekey> [sourcekey …] 将一个或多个HLL合并后的结果存储在另一个HLL中，比如每月活跃用户可以使用每天的活跃用户来合并计算可得

实例:

​ 即使用pfadd命令将每次访问的用户id存入,需要统计是使用pfcount,如果需要统计多天的数据使用pfmerge

5.3 Geospatial

专门用以经纬度的数据.类型,有经纬度设置,经纬度查询,范围查询,距离查询,经纬度Hash等

geoadd<key><longitude><latitude><member> [longitude latitude member…] 添加地理位置(经度，纬度，名称)

geopos <key><member> [member…] 获得指定地区的坐标值

geodist<key><member1><member2> [m|km|ft|mi ] 获取两个位置之间的直线距离,m米,km千米,ft英尺,mi英里,默认为mi

georadius<key><longitude><latitude>radius m|km|ft|mi 以给定的经纬度为中心，找出某一半径内的元素

5.4 Streams

是一种仅允许追加的日志格式,

  1. XADD key [NOMKSTREAM(默认情况下key不存在会新建,添加后则不会新建)] [MAXLEN|MINID [=|~] threshold [LIMIT count]] *|ID field value [field value …]

    添加Redis自动生成Id

    添加手动输入Id

    返回的是该组元素在此Stream下的ID.如果是Redis自动生成的Id,则有两个部分:<millisecondsTime>-<sequenceNumber>,序列号部分的长度为64位.

    需要注意,不能添加Id值小于或等于已存在Id的元素

  2. XLEN <key> 查看该key下有多少组元素,每一次Add为一组

  3. XRANGE|XREVRANGE key start end [COUNT count] 查看该key中,id从start到end中的所有数据,- + 代表总最小的到最大的

6.事务

Redis事务中的所有命令都会被序列化.按照顺序的执行.事务在执行过程中,不会被其他命令所打断,故主要作用是串联多个命令防止被别的命令插队.

通过Multi开启事务,Exec执行事务,Discard放弃事务,Watch开启CAS,Unwatch关闭所有key的CAS

事务中的错误可以分为两种

  1. 在exec命令执行前的报错,比如命令关键词错误,参数数量不正确等,这种错误整个事务中的命令都不会执行
  2. 在exec命令执行后出现的错误,例如对一个String类型的key使用了List的命令,这种错误不会导致事务失效,其余命令都会正常执行

对于watch,在multi前开启watch可以监视指定key,在事务过程中被监视的key的值发生变化则会导致整个事务失效,但是要注意,一旦exec后,不管事务执行成功与否,所有watched的key都会变成unwatched的状态.

7.持久化

7.1 RDB(Redis Database)

按照指定的时间间隔将内存中的数据集快照写入磁盘,恢复时直接将快照读到内存中

备份执行过程:

  1. Redis单独创建线程来进行持久化

  2. 先将数据写入到一个临时文件中

  3. 临时文件写入完成后使用该临时文件替换上次的文件

    整个过程中,Redis主线程不参与IO,保证性能,如果对于数据恢复的完整性要求不高,RDB会比较好.

    缺点是最后一次持久化后新增的数据可能丢失

7.2 AOF(Append Only File)

以日志的形式记录每一个写操作,只允许追加文件内容不允许修改文件内容.

备份执行过程:

  1. 写命令追加至AOF缓存区

  2. AOF缓冲区根据AOF持久化策略(always,everysec,no)将操作sync到磁盘中

  3. AOF文件大小超过重写策略或者手动重写时,对AOF文件重写,压缩空间

    AOF与RDB同时开启时,默认使用AOF的方式恢复数据

8.应用问题

8.1 缓存穿透

key对应的数据在数据源并不存在，每次针对此key的请求从缓存获取不到，请求都会压到数据源，从而可能压垮数据源。

解决方案

  1. 对空值缓存:即使数据源每次查询后的结果为空,也依然将该空值缓存至redis,只是过期时间设置较短

  2. 如果是根据Id查询,可以将所有Id放入set中,没有命中缓存时,可以与set中的Id进行对比,如果不存在直接返回

  3. 设置黑名单,对短时间内多次访问的Id加入黑名单,一定时间内不允许访问

  4. 布隆过滤器:

    对于元素a,b,c,存入时需要经过三次Hash运算.比如a元素,需要f1(a),f2(a),f3(a)三次运算,并将运算结果上的值修改为1,那么当需要判断a元素是否存在时,也是通过计算f1(a),f2(a),f3(a)三个位置然后判断三个位置上的值是否都为1,如果都为1则a存在,反之则不存在.

    如果元素不存在,布隆过滤器有可能判断存在;如果元素存在,布隆过滤器一定判断存在

    使用Bitmaps实现 https://zhuanlan.zhihu.com/p/150333265

8.2 缓存击穿

Redis中的热点数据过期,大量请求访问数据库,数据库宕机(针对一个key)

解决方案:

  1. 在Redis高峰前,提前设置热点数据,并设置较长的过期时间
  2. 加锁,允许个别线程去进行load db的操作,其余线程等待一段时间后再次访问

8.3 缓存雪崩

Redis中大量数据同时过期,大量请求访问数据库,数据库宕机(针对多个key)

解决方案:

  1. 设置多级缓存,nginx缓存+redis缓存+其他缓存
  2. 加锁,允许个别线程去进行load db的操作,其余线程等待一段时间后再次访问
  3. 记录缓存的过期时间,在缓存过期前提前进行load db的操作
  4. 将缓存的过期时间设置为一个随机值,防止大量缓存同时过期

8.4 缓存倾斜问题

热点数据都在一台Redis节点上,Redis无法承受大量请求,Redis宕机

解决方案:

  1. 主从架构,搭设多个从节点,缓解主节点压力
  2. 在redis前设置缓存,缓解redis压力
